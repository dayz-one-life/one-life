## 1. Goals

- Light up the reserved `articles.image_url` / `image_prompt` / `image_kind` columns (R5a migration
  `0009`) for both article kinds, **including backfill** of the existing catalogue.
- One image per article, `image_kind = 'hero'` (`card`/`breaking` stay reserved for future
  verticals).
- Feed-level visual variety: no two adjacent covers with the same framing.
- Durability equal to the prose: the articles table carries a permanent-archive promise, so image
  bytes live in Postgres and ride the existing pg_dump backup.
- Same operational posture as the rest of the newsdesk: dry-run by default, bounded attempts,
  failure-isolated, no live model calls in tests.

## 3. Art direction â€” brand-bible Â§10.4, vendored

Source of truth: `../brand/brand-bible.md` Â§10.4 + `../brand/content-engine/image-prompt-scaffold.md`
and `image_prompt.py`. Following the R5a voice.ts precedent (and the ratified "vendored, no
cross-repo build dependency" rule â€” **no submodule**), the scaffold is ported verbatim to a new
`apps/newsdesk/src/image-prompt.ts` with a source-of-truth pointer comment:

- `IMAGE_STYLE` â€” the fixed style signature, appended to every prompt: *"Shot on a cheap 1990s
  point-and-shoot film camera with direct on-camera flash: harsh flat frontal flash, blown-out
  highlights, hard black shadows, heavy film grain, slight motion blur, candid and unposed, awkward
  off-center crop, authentic amateur tabloid photojournalism, gritty weathered realistic textures,
  damp overcast Eastern-European wilderness. Photorealistic, imperfect, real."*
- `IMAGE_ANTISLOP` â€” the fixed anti-slop clause, appended to every prompt: *"It must NOT look like:
  a video game, a 3D render, CGI, an illustration, digital art, concept art, a movie still,
  cinematic or dramatic studio lighting, HDR, bokeh, smooth airbrushed skin, glamorous, beautiful,
  symmetrical, hyperdetailed, or professionally composed."*
- `buildImagePrompt(scene, kind)` â€” `"{scene}\n\n{STYLE}\n\n{ANTISLOP}\n\nAspect ratio {ratio}."`
  with the Â§10.4 ratio table (`hero` 4:5, `card` 1:1, `breaking` 16:9); throws on unknown kind.

Hard rules that ride in the scene-writer's system prompt regardless of category:

- **Imply-don't-depict:** never a corpse or gore; death is aftermath, absence, objects, witnesses.
  (Taste rule, ethics rule, and content-filter workaround in one â€” validated: cheap models refuse
  corpses but generate aftermath cleanly.)
- **Fog Rule:** generic unidentifiable locales only; living subjects (birth notices) stay deniable
  (distant/obscured/blurred). No landmarks, no base layouts.
- No legible text, no logos, no real-person likenesses; imperfect or partially obscured faces are a
  feature. One clear subject mid-moment; never posed, never glamorous.
- Tone: Morgue = deadpan mock-gravity; Nursery = doomed optimism. Punch up, never down; rib
  first-lifers affectionately.

**Aspect reality (verified by live probe):** the gpt-image models expose no aspect/size/format
parameters on the unified API and return a ~1024Ă—1024 PNG regardless of prompt text (one sample
came back 1024Ă—1536 unprompted). The "Aspect ratio 4:5." line stays (composition nudge); the 4:5
hero and 1:1 thumb are **render-side crops** (`object-fit: cover`).

## 4. Scene variety â€” menu as rails, LLM as author, recency as memory

Per image, a **scene-writer LLM call** (the existing `CompletionClient`, `NEWSDESK_MODEL`) receives:
the article's stored `facts` + headline + lede (the story's voice), the category menu
**pre-filtered in code to facts-eligible entries**, the escape hatch, and a **do-not-repeat block**
of the last 8 same-kind covers (caption + scene line). It returns `{caption, scene}` JSON â€”
zod-validated (caption 3â€“48 chars, uppercased server-side; scene 20â€“600 chars, single paragraph).

- **Escape hatch:** the LLM should prefer the menu but may invent a category when the story earns
  it, supplying its own caption. The hard rules bind custom categories identically (they're framing
  inventions, not rule exemptions).
- **Recency exclusion:** the scene line is the first paragraph of the stored `image_prompt` (the
  text before `IMAGE_STYLE`), so the do-not-repeat list needs no extra storage.
- **Gating:** a pure `eligibleCategories(kind, facts)` evaluates each entry's predicate against the
  stored `facts` jsonb; the LLM never sees an ineligible category.

## 5. Data model (migration `0012`)

`articles` (columns `image_url`/`image_prompt`/`image_kind` already exist from `0009`):

| column | type | notes |
|---|---|---|
| `image_caption` | text, null | the deadpan caps line; stored text so menu + escape-hatch captions render identically |
| `image_model` | text, null | provenance, mirrors `model` |
| `image_attempts` | integer, not null, default 0 | image-pass retries, independent of text-gen `attempts` |
| `image_error` | text, null | last image-pass failure |

New table **`article_images`** (durable; in `APP_TABLES`; excluded from `--rebuild` truncation,
like `articles`):

| column | type | notes |
|---|---|---|
| `article_id` | bigint PK, FK â†’ `articles.id` on delete cascade | one image per article |
| `bytes` | bytea, not null | the original model output, unmodified |
| `content_type` | text, not null | from the API response `media_type` (png observed; jpeg/webp possible) |
| `width` / `height` | integer, null | parsed from the PNG IHDR when png; null otherwise |
| `created_at` | timestamptz, not null, default now | |

`image_url` stores `/media/heroes/<slug>.<ext>` (ext from `content_type`), so read models, API,
web, OG, and JSON-LD all consume a plain URL string. Published articles always have a slug (only
failed stubs don't, and the pass targets `status='published'` only). A partial index
`articles_image_missing_idx` on `(created_at desc) WHERE status='published' AND image_url IS NULL`
keeps the target scan cheap as the catalogue grows.

## 6. Newsdesk image pass

A fourth sibling in the `main.ts` loop (after obituary, birth, Discord notify), own try/catch, same
`{generated, failed, skipped, dryRun}` result shape.

Flow per tick (`imageTick(db, deps)`):

1. `findImageTargets(db, {limit: batchCap, maxAttempts})` â€” `status='published' AND image_url IS
   NULL AND image_attempts < maxAttempts`, both kinds, **`created_at` desc** (fresh articles jump
   the queue; the backfill drains behind at `batchCap` per tick â€” with defaults, ~10 images / 5
   min).
2. Per target: build the recency list (last 8 same-kind published rows with `image_url NOT NULL`,
   newest first â†’ caption + scene line); `eligibleCategories(kind, facts)`;
   **dry-run gate here** â€” log `would generate image` with slug + eligible-category count, no
   client calls, no writes.
3. Scene-writer call â†’ `{caption, scene}` (zod). 4. `buildImagePrompt(scene, 'hero')`.
5. `ImageClient.generate({prompt, model, quality})` â€” model = `NEWSDESK_IMAGE_MODEL_FLAGSHIP` when
   the stored facts say `isLegend` (obituaries), else `NEWSDESK_IMAGE_MODEL`.
6. Persist in one transaction: insert `article_images`, update the article
   (`image_url`, `image_prompt` = full assembled prompt, `image_kind='hero'`, `image_caption`,
   `image_model`, `image_attempts+1`, `image_error=null`).
   On any step failing: `image_attempts+1` + `image_error`; at `maxAttempts` the article stays
   text-only permanently (every surface degrades gracefully).

**`ImageClient`** is a new injected seam beside `CompletionClient`:
`{ generate(req: {prompt, model, quality}): Promise<{bytes, contentType}> }`. The OpenRouter
implementation POSTs `https://openrouter.ai/api/v1/images` with `{model, prompt, quality, n: 1}`
and base64-decodes `data[0].b64_json` / reads `data[0].media_type`; throws on error payloads,
missing data, or non-2xx. (The response is always base64 bytes â€” OpenRouter hosts nothing.)

Config (`config.ts`, following house patterns):

| env | default | notes |
|---|---|---|
| `NEWSDESK_IMAGE_MODEL` | `openai/gpt-5-image-mini` | Â§10.4's validated workhorse |
| `NEWSDESK_IMAGE_MODEL_FLAGSHIP` | `openai/gpt-5.4-image-2` | legends' send-offs |
| `NEWSDESK_IMAGE_QUALITY` | `low` | validated: ~$0.003/image, quality held |
| `NEWSDESK_IMAGES_ENABLED` | `true` | independent kill-switch; a broken image pipeline must never stop the prose |

Shared gates reused as-is: `NEWSDESK_DRY_RUN` (default true â€” the pass is inert until go-live),
`NEWSDESK_BATCH_CAP`, `NEWSDESK_MAX_ATTEMPTS`, `OPENROUTER_API_KEY`.

**No separate image prompt-version:** the full prompt is persisted verbatim in `image_prompt`,
which is stronger provenance than a version string.

**Cost (measured, 2026-07-17):** scene call â‰ˆ $0.001 + image at quality `low` â‰ˆ $0.003 â†’ **~$0.004
per article**. The 11-category validation run: 11/11 first-attempt successes, zero content-filter
refusals, $0.030 total.

## 7. API â€” the `/media` route lights up

## 11. Risks & honest limitations

- **Model behavior drift:** aspect and refusal behavior were probed on 2026-07-17; a provider
  change could reintroduce refusals â€” bounded attempts + `image_error` make that visible, and the
  kill-switch stops spend without stopping prose.
- **Escape-hatch quality:** a custom caption can be mediocre; zod bounds format, not wit. Accepted â€”
  variety is the point; menu-only remains a one-line prompt change.
- **Square sources, portrait crops:** 4:5/1:1 crops of a square can clip composition edges; Â§10.4's
  "awkward off-center crop" makes this stylistically load-bearing rather than a defect.
- **DB growth:** ~2 MB/article (~2 GB per thousand). Accepted for the archive promise; if it ever
  hurts, a later slice can move bytes out behind the same URL contract.
- **`next/image` optimizer** becomes a runtime dependency of article pages (first use in repo); its
  cache lives on the web host's disk.
